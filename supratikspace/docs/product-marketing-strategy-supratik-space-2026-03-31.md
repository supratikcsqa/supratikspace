# Product Marketing Strategy: Supratik Space Builder Lite

## 1. Executive Summary
Supratik.space should not market itself as a futuristic control centre. It should market itself as the fastest way for a busy builder to turn live work into a personal site, publish open-source projects on clean subdomains, and launch under a custom domain without maintaining a full website stack. The recommended sequence is simple: reposition the homepage first, launch the GitHub-subdomain wedge second, and attach the $10 template-clone offer as the monetization layer.

## 2. ICP and Buyer Map
**Primary ICP:** indie hackers, open-source maintainers, developer-creators, and productized freelancers who already ship work but do not want to handcraft a marketing site.  
**Buyer pains:** stale portfolios, fragmented profile links, weak project presentation, and too much time spent maintaining personal branding.  
**Switching triggers:** a new project launch, job search, founder intro, demo day, or a desire to look more credible online fast.  
**Decision style:** low patience, high taste, outcome-driven, willing to spend a small amount for speed and clarity.

## 3. Market Problem and Timing
Builders already have options, but the current market splits in awkward ways:

- **GitHub Pages** gives builders hosting directly from repositories, but not a polished personal-site or monetized template experience. Source: [pages.github.com](https://pages.github.com/)
- **Framer** has strong portfolio templates and visual polish, but it does not own the "live open-source builder identity" wedge. Source: [framer.com/templates/category/portfolio](https://www.framer.com/templates/category/portfolio/)
- **Super** makes fast websites from Notion and supports custom domains, but the core story is content publishing rather than builder-native project showcase. Source: [super.so](https://super.so/)
- **about.me** offers a personal page plus creator/business features, but it is not built around live GitHub work or project-level hosting. Source: [about.me](https://about.me/)
- **Typedream** is fast and creator-friendly, but it is broader than the sharp builder persona you can own. Source: [typedream.com](https://typedream.com/)

The gap is a product that says: "show the work you are already shipping, not a static bio you now have to maintain."

## 4. Positioning Statement
Supratik.space helps busy builders turn their live work into a personal website in minutes, so they can showcase projects, open source, and credibility without learning another website builder.

## 5. Key Messages and Proof Points
1. **Your work becomes the website.** Proof point: GitHub import and live project subdomains make the site feel current, not manually curated.
2. **You ship fast, not from scratch.** Proof point: a single template can be configured for resume, portfolio, or project showcase modes.
3. **You own the outcome, not the maintenance burden.** Proof point: optional source connections and guided custom-domain mapping reduce upkeep.
4. **The offer is low-risk.** Proof point: a $10 entry price makes the first step feel lightweight and easy to try.

## 6. Offer, Packaging, or Pricing Hypotheses
- **Core offer:** one-time $10 starter purchase for a cloned personal website, hosted on supratik.space with a simple control panel.
- **Scope note:** for the lightweight release, "full CRM" should be interpreted as a builder control panel for profile sources, featured work, links, and publishing state. A true CRM is a later-stage product.
- **Optional upgrade path:** custom domain connection as part of the same starter flow or as a small follow-on service.
- **Template packaging:** sell outcomes, not template count. Frame one template as three use-case modes until more designs exist.

## 7. Launch Plan and Channel Strategy
1. **Landing page relaunch on March 31, 2026:** this is the highest-leverage update because every later feature depends on relevance and trust at first glance.
2. **GitHub-subdomain beta on April 1, 2026:** invite builders to submit a public repo or profile and receive a live preview. This fits the ICP because GitHub is already their source of truth.
3. **$10 publish offer on April 2, 2026:** add a low-friction purchase action once previews exist. The purchase should feel like publishing, not buying software.
4. **Founder-led distribution:** announce on X, LinkedIn, GitHub README links, project footers, and builder communities because the ICP already follows working builders there.
5. **Concierge onboarding for first customers:** keep the first cohort high-touch so product decisions are shaped by real objections and setup pain.

## 8. Funnel and Conversion Metrics
- **Top of funnel:** landing page CTA click-through rate, use-case selection rate, template preview starts.
- **Mid funnel:** GitHub import completion, preview generation success, profile source connections.
- **Bottom of funnel:** preview-to-paid conversion, publish completion, custom-domain completion.
- **Retention:** weekly active published-site owners and repeat dashboard updates after launch.

## 9. Sales Enablement Assets Needed
- A homepage hero and subhead that explain the product in one read.
- A template showcase block framed by outcomes: resume, portfolio, project showcase.
- A short demo clip or GIF showing repo import to live subdomain.
- A clear FAQ covering domains, hosting, what "$10" includes, and what "dashboard" means.
- A founder script for DMs, posts, and community replies that emphasizes speed and live work.

## 10. Risks, Assumptions, and Open Questions
- **Risk:** "Any GitHub project can be hosted" can imply full deployment, which is materially heavier than project showcasing. The product copy must define whether this means a project page, a proxied static build, or both.
- **Risk:** calling the dashboard a "full CRM" raises expectations far beyond a lightweight release.
- **Risk:** one template can still work, but only if the framing is strong and the content blocks adapt by use case.
- **Assumption:** the first cohort values speed and clarity more than unlimited customization.
- **Open question:** does the first release optimize for open-source maintainers, freelance builders, or job-seeking developers first? The homepage can speak to all three, but the hero should choose one lead persona.
